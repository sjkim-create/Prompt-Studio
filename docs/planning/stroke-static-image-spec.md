# 필기 획 순서·딜레이를 정적 이미지로 인코딩하기 — 기술 기획

## 배경 및 목적

### 문제

수학 풀이 과정을 채점할 때, 현재는 **동영상**을 LLM(Gemini 등)에 넣어야만 풀이 과정(획 순서, 사고 흐름)을 분석할 수 있다. 이는:

- 동영상 지원 LLM에만 의존 (Claude, GPT-4o 등에서는 동영상 불가)
- 토큰 비용이 크다
- 처리 속도가 느리다

### 목표

**1~2장의 PNG 이미지 + 간단한 텍스트 설명**만으로, 동영상이 담고 있던 다음 정보를 LLM에 전달한다:

1. **획의 순서** — 어떤 글자/기호를 먼저 쓰고 나중에 썼는지
2. **획 간 딜레이** — 망설임, 사고 시간, 수정 등의 시간적 패턴

채점 맥락에서 이 정보가 왜 중요한가:
- 풀이 순서가 올바른지 (예: 괄호 먼저 계산했는지)
- 지우고 다시 쓴 부분이 있는지 (긴 딜레이 후 같은 위치에 다시 쓰기)
- 답을 먼저 쓰고 풀이를 나중에 채운 것은 아닌지

## 실제 데이터 분석 (초등 수학 문제 1)

| 항목 | 값 |
|------|-----|
| 총 획 수 | 120 |
| 총 시간 | 65.3초 |
| 딜레이 최소 | 21ms |
| 딜레이 최대 | 5,326ms (5.3초) |
| 딜레이 중앙값 | 108ms |

### 딜레이 분포

| 구간 | 횟수 | 의미 |
|------|------|------|
| < 100ms | 53 (45%) | 같은 글자 내 획 연결 |
| 100~300ms | 46 (39%) | 글자 간 자연스러운 전환 |
| 300~500ms | 6 (5%) | 짧은 생각/확인 |
| 500ms~1s | 2 (2%) | 의미 있는 멈춤 |
| 1~2s | 9 (8%) | 사고/계산 시간 |
| 2~5s | 2 (2%) | 긴 사고 시간 |
| > 5s | 1 (1%) | 매우 긴 멈춤 (섹션 전환) |

**핵심 인사이트**: 딜레이의 약 84%는 300ms 미만이며 필기 흐름의 자연스러운 부분이다. 채점에 의미 있는 딜레이는 **500ms 이상** (약 12%, 14건)으로, 이것만 표시하면 충분하다.

## 방안 비교

### 방안 A: 단일 이미지 — 색상 순서 + 딜레이 마커 ⭐ 권장

1장의 이미지에 모든 정보를 담는다.

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   [메인 캔버스]                                     │
│                                                     │
│   각 획이 순서에 따른 색상으로 그려짐               │
│   빨강(#1) → 주황 → 노랑 → 초록 → 파랑 → 보라(#N)   │
│                                                     │
│   500ms 이상 딜레이 지점에 마커:                    │
│   해당 획의 시작점에  ○ 1.2s  형태로 표시           │
│                                                     │
│                                                     │
├─────────────────────────────────────────────────────┤
│ [색상바]  ■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■          │
│           #1 (첫 획)            #120 (마지막 획)  │
├─────────────────────────────────────────────────────┤
│ [타임라인]                                          │
│ |■■■|─0.8s─|■■|──1.5s──|■■■■■|──5.3s──|■■■|         │
│  획 블록(색상)  딜레이(빈 공간+라벨)                │
│                                                     │
│ 총 획: 120 | 총 시간: 65.3s | ○=멈춤(≥0.5s)         │
└─────────────────────────────────────────────────────┘
```

**장점**: 토큰 최소, LLM이 색상+텍스트 모두 인식 가능
**단점**: 색상이 원본 필기의 검은색을 대체하므로 LLM이 글자를 읽기 어려울 수 있음

### 방안 B: 2장 이미지 — 원본 + 메타 오버레이 ⭐⭐ 채점용 최적

**이미지 1: 원본 필기** (기존 page.png와 동일)
- 검은색 획으로 원래대로 렌더링
- LLM이 "수학 풀이"로 인식하고 내용을 읽는 용도

**이미지 2: 메타데이터 이미지**
- 방안 A와 동일하지만, LLM에게 "이미지 1의 획 순서/딜레이 정보"라고 설명

**장점**: 원본 가독성 유지 + 메타데이터 분리
**단점**: 이미지 2장 = 토큰 약 2배

### 방안 C: 단일 이미지 — 좌우/상하 분할

```
┌───────────────────┬───────────────────┐
│ [원본 필기]       │ [순서 색상 버전]  │
│ 검은색            │ 컬러 + 딜레이 마커│
├───────────────────┴───────────────────┤
│ [범례 + 타임라인]                     │
└───────────────────────────────────────┘
```

**장점**: 1장에 원본과 메타 모두 포함
**단점**: 해상도 반감, 복잡한 필기에서 가독성 저하

## 딜레이 인코딩: 시간 비례 색상바

### 핵심 아이디어

색상바의 가로 축을 **시간에 비례**시킨다. 단순히 획 번호 순으로 균등 분할하는 것이 아니라, 각 획의 소요 시간 + 획 간 딜레이를 반영하여 색상이 차지하는 너비를 결정한다.

### 균등 분할 vs 시간 비례 비교

**기존 (균등 분할)** — 순서만 표현, 딜레이 정보 없음:
```
|빨|주|노|초|파|보|     ← 각 획이 동일 너비
 #1 #2 #3 #4 #5 #6
```

**개선 (시간 비례)** — 순서와 딜레이를 동시에 표현:
```
|빨빨빨빨빨|주|노노|초초초초초초초초|파파|보|
 #1(오래)   #2 #3   #4(앞에 긴 딜레이) #5  #6
```

이 방식에서:
- 색상이 넓게 차지 = 해당 획이 오래 걸렸거나, 다음 획까지 딜레이가 길었음
- 색상이 빠르게 전환 = 빠르게 연속 필기
- LLM이 "색상 전환 속도"를 시각적으로 직관 파악 가능

### 구현 방식

색상바의 각 픽셀이 일정 시간(ms)을 나타내도록 한다:

```typescript
// 색상바 각 위치의 색상 계산
function getColorAtTime(timeMs: number, strokes: Stroke[]): string {
  // timeMs 시점에 활성인(또는 직전에 완료된) 획의 index를 찾는다
  const strokeIndex = findStrokeAtTime(timeMs, strokes);
  return strokeColor(strokeIndex, strokes.length);
}
```

각 획이 "소유"하는 시간 구간:
- 획 i의 필기 구간 = `[stroke[i].startTime, stroke[i].endTime]`
- 딜레이 구간 = `[stroke[i].endTime, stroke[i+1].startTime)` (이전 획의 **끝 시간**과 다음 획의 **시작 시간** 차이)
- 딜레이 시간 동안은 **직전 획의 색상이 계속 유지**됨
- ⚠️ 딜레이는 반드시 `stroke[i+1].startTime - stroke[i].endTime`으로 계산해야 한다. 시작 시간끼리 비교하면 획 자체의 소요 시간이 딜레이에 포함되어 부정확해진다.

### 왜 이 방법이 효과적인가

1. **별도 타임라인 불필요** — 색상바 하나로 순서 + 딜레이 모두 표현
2. **직관적** — "같은 색이 길게 늘어져 있다" = "여기서 오래 멈췄다"
3. **정보 밀도 높음** — 추가 공간 소비 없이 기존 색상바에 시간 정보 통합
4. **LLM 친화적** — 색상 전환 패턴은 LLM이 시각적으로 쉽게 인식

### 보조 수단: 큰 딜레이에 눈금 + 라벨

시간 비례 색상바만으로는 정확한 초 단위를 알기 어려우므로, 큰 딜레이(≥500ms) 지점에 눈금과 시간 라벨을 추가한다:

```
|빨빨빨빨빨|주|노노|          초초초초초초초초          |파파|보|
                   ↑ 5.3s
0s        8s  10s  12s                        27s     32s  35s
```

### 메인 캔버스의 딜레이 마커

색상바와 별개로, 메인 캔버스에서도 큰 딜레이(≥500ms)는 해당 획 시작점에 텍스트 마커(`● 1.2s`)로 표시한다. LLM은 이미지 내 텍스트를 가장 정확하게 읽으므로, 정확한 수치가 필요한 경우를 위한 이중 안전장치이다.

## 최종 권장안: 방안 A 확장 (단일 이미지)

채점 용도에서는 원본 필기(page.png)가 이미 별도로 존재하므로, 메타데이터 이미지 1장만 추가하면 된다.

### 이미지 레이아웃 상세

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│   [A. 메인 캔버스]                                       │
│                                                          │
│   • 각 획을 HSL 색상 그라디언트로 렌더링                 │
│     hsl(순서비율 × 270, 85%, 45%)                        │
│     빨강(0°) → 주황(45°) → 노랑(90°) → 초록(135°)        │
│     → 시안(180°) → 파랑(225°) → 보라(270°)               │
│                                                          │
│   • 딜레이 ≥ 500ms인 지점에 마커:                        │
│     작은 원 ● + "1.2s" 텍스트                            │
│     원의 크기: 딜레이에 비례 (500ms=작은원, 5s=큰원)     │
│                                                          │
│   • 각 획의 시작점에 작은 점(dot)으로 시작 위치 표시     │
│                                                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│   [B. 시간 비례 색상바] (순서 + 딜레이 통합)             │
│                                                          │
│   |■■■■■■|■■|■■|          ■■■■■■■■■■          |■■■|■|    │
│    빨강     주황 노랑  ↑5.3s  초록           파랑  보라  │
│   0s       8s  10s  12s                 27s   32s  35s   │
│                                                          │
│   가로축 = 시간, 색상 = 획 순서                          │
│   같은 색이 넓음 = 다음 획까지 딜레이가 길었음           │
│   ≥0.5s 딜레이 지점에 눈금+라벨 표시                     │
│                                                          │
├──────────────────────────────────────────────────────────┤
│                                                          │
│   [D. 요약 정보]                                         │
│                                                          │
│   총 획: 120 | 총 시간: 65.3s                            │
│   의미 있는 멈춤(≥0.5s): 14회                            │
│   최대 멈춤: 5.3s (획 #72~#73 사이)                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 색상 함수

```typescript
function strokeColor(index: number, total: number): string {
  const hue = (index / Math.max(total - 1, 1)) * 270;
  return `hsl(${hue}, 85%, 45%)`;
}
```

왜 HSL 270도까지만 쓰는가:
- 360도를 쓰면 처음(빨강)과 끝(빨강)이 구분 불가
- 270도: 빨강 → 보라까지, 처음과 끝이 명확히 구분됨
- 85% 채도, 45% 명도: 흰색 배경에서 모든 색상이 선명

### 딜레이 임계값

| 딜레이 | 마커 | 이유 |
|--------|------|------|
| < 300ms | 무시 | 자연스러운 필기 흐름 (전체의 84%) |
| 300~499ms | 작은 점만 | 짧은 확인/멈춤 |
| 500ms~2s | ● + 텍스트 `0.5s` | 사고/계산 시간 |
| ≥ 2s | ●● + 굵은 텍스트 `5.3s` | 긴 사고, 섹션 전환 |

### 딜레이 마커 겹침 방지

획이 밀집된 영역에서 마커가 겹칠 수 있다. 해결:

1. **오프셋**: 마커가 겹치면 위/아래로 약간 이동
2. **리더 라인**: 마커를 화살표로 해당 획 시작점에 연결
3. **최후 수단**: 겹치는 마커는 하단 요약 테이블로만 표시

## LLM 프롬프트 설계

### 채점 시 이미지와 함께 보낼 텍스트 (약 80 토큰)

```
[첨부: 원본 필기 이미지, 획 분석 이미지]

획 분석 이미지 설명:
- 획의 색상 = 작성 순서 (빨강=처음, 보라=마지막, 하단 색상바 참조)
- ● + 숫자 = 해당 위치에서 멈춘 시간(초)
- 하단 타임라인 = 전체 필기 흐름 (획 블록 + 멈춤 간격)
이 정보를 참고하여 풀이 과정의 순서가 올바른지 채점하세요.
```

### 토큰 효율 분석

| 방식 | 예상 입력 토큰 |
|------|---------------|
| 동영상 (65초) | ~6,500 토큰 (Gemini 기준 초당 ~100) |
| 이미지 2장 + 텍스트 | ~1,700 토큰 (이미지 ~800×2 + 텍스트 80) |
| 이미지 1장 + 텍스트 | ~900 토큰 (이미지 ~800 + 텍스트 80) |

**약 7배의 토큰 절감** (동영상 대비 단일 이미지)

## 대안: 텍스트만으로 표현 (이미지 없이)

만약 토큰을 극단적으로 줄이고 싶다면, 획 메타데이터를 **구조화된 텍스트**로도 표현 가능:

```
획 순서 및 딜레이:
#1-#15 (0~8s): "1. 계산순서" 제목 + 수식 작성
  [1.2s 멈춤]
#16-#45 (9~22s): 계산 순서 표시 (①②③④ + 연결선)
  [5.3s 멈춤]
#46-#55 (27~32s): "2. 풀이과정 및 답" 제목
#56-#120 (32~65s): 풀이 과정 작성
```

그러나 이 접근은 **획의 공간적 위치**가 사라지므로, "어떤 부분을 먼저 썼는지"를 LLM이 원본 필기와 대응시키기 어렵다. 이미지 기반이 더 효과적.

## 구현 계획

### 컴포넌트 구조

```
src/
  StrokeImageGenerator.tsx    — 메타데이터 이미지 생성 컴포넌트
  utils/
    strokeColor.ts            — HSL 색상 계산
    delayAnalysis.ts          — 딜레이 분석 및 임계값 판정
    timelineRenderer.ts       — 하단 타임라인 렌더링
```

### 핵심 구현 항목

1. **순서 색상 렌더링**: 각 획을 `strokeColor(i, total)`로 그림
2. **딜레이 마커**: 500ms 이상 딜레이에 ● + 텍스트 배치
3. **색상 범례 바**: 그라디언트 바 + 눈금
4. **타임라인**: 가로 막대그래프 형태, 획=색상 블록, 딜레이=빈 공간
5. **요약 정보**: 텍스트로 통계 표시
6. **PNG 다운로드**: `canvas.toBlob()` → 다운로드

### 이미지 사양

- 해상도: 1200×1700px (A4 비율 근사)
- 메인 캔버스 영역: 1200×1200px
- 범례+타임라인+요약: 1200×500px
- 배경: 흰색 (#FFFFFF)
- 출력 포맷: PNG (투명도 없음)

## 검증 계획

1. 생성된 이미지를 Claude/GPT-4o에 넣고 "이 필기의 작성 순서를 설명하세요"라고 질문
2. LLM이 색상→순서 매핑을 정확히 해석하는지 확인
3. 딜레이 마커의 값을 정확히 읽는지 확인
4. 타임라인에서 전체 흐름을 파악하는지 확인

### 토큰 비용 비교: 동영상 vs 이미지

LLM의 동영상 처리 토큰 비용은 모델마다 다르지만, 대략적인 추정치는 다음과 같다:
- **Gemini**: 초당 ~258 토큰 (공식 문서 기준, 1초 = 258 토큰)
- **이미지 1장**: 해상도에 따라 다르나 800×1400 이미지 기준 약 1,100~1,600 토큰
- **프롬프트 텍스트**: 채점 프롬프트 약 1,500 토큰 + 이미지 해석 가이드 약 500 토큰

#### 1문제당 1분 풀이 시나리오

| 입력 방식 | 토큰 구성 | 총 토큰 (추정) |
|----------|----------|---------------|
| **동영상 (60초)** | 60s × 258 = 15,480 + 프롬프트 1,500 | **~17,000** |
| **A안 (1장)** | 이미지 1,400 + 프롬프트 2,000 | **~3,400** |
| **B안 (2장)** | 이미지 1,400×2 + 프롬프트 2,000 | **~4,800** |

→ A안은 동영상 대비 **약 5배 절감**, B안은 **약 3.5배 절감**

#### 1문제당 2분 풀이 시나리오 (중등 문제 수준)

| 입력 방식 | 토큰 구성 | 총 토큰 (추정) |
|----------|----------|---------------|
| **동영상 (120초)** | 120s × 258 = 30,960 + 프롬프트 1,500 | **~32,500** |
| **A안 (1장)** | 이미지 1,400 + 프롬프트 2,000 | **~3,400** |
| **B안 (2장)** | 이미지 1,400×2 + 프롬프트 2,000 | **~4,800** |

→ A안은 동영상 대비 **약 9.6배 절감**, B안은 **약 6.8배 절감**

#### 1문제당 5분 풀이 시나리오 (고등 문제 또는 복잡한 서술형)

| 입력 방식 | 토큰 구성 | 총 토큰 (추정) |
|----------|----------|---------------|
| **동영상 (300초)** | 300s × 258 = 77,400 + 프롬프트 1,500 | **~78,900** |
| **A안 (1장)** | 이미지 1,400 + 프롬프트 2,000 | **~3,400** |
| **B안 (2장)** | 이미지 1,400×2 + 프롬프트 2,000 | **~4,800** |

→ A안은 동영상 대비 **약 23배 절감**, B안은 **약 16배 절감**

### 요약: 동영상 대비 절감률

| 풀이 시간 | 동영상 토큰 | A안 토큰 | A안 절감률 | B안 토큰 | B안 절감률 |
|----------|-----------|---------|----------|---------|----------|
| 1분 | ~17,000 | ~3,400 | **5.0x** | ~4,800 | **3.5x** |
| 2분 | ~32,500 | ~3,400 | **9.6x** | ~4,800 | **6.8x** |
| 5분 | ~78,900 | ~3,400 | **23.2x** | ~4,800 | **16.4x** |

**핵심**: 이미지 방식은 풀이 시간에 관계없이 토큰이 일정하다. 동영상은 시간에 비례하여 토큰이 증가하므로, **풀이가 길수록 이미지 방식의 절감 효과가 극적으로 커진다.**

### 기능 비교: 동영상이 가지는 정보 vs 이미지가 가지는 정보

| 정보 | 동영상 | A안 (1장) | B안 (2장) |
|------|--------|----------|----------|
| 필기 내용(글자, 수식) 읽기 | ✅ 좋음 | ⚠️ 색상 때문에 가독성 저하 | ✅ 원본 이미지에서 명확 |
| 획 순서 | ✅ 시간 흐름으로 자연스럽게 파악 | ✅ 색상으로 파악 | ✅ 색상으로 파악 |
| 획 간 딜레이(사고 시간) | ✅ 실시간 멈춤으로 직관적 | ✅ 마커 + 시간 비례 색상바 | ✅ 마커 + 시간 비례 색상바 |
| 수정/지우기 흔적 | ✅ 동영상에서 실시간으로 보임 | ⚠️ 같은 위치 다른 색상으로 추론 | ⚠️ 같은 위치 다른 색상으로 추론 |
| 필기 속도 변화 | ✅ 빠르게/느리게 쓰는 것 직관적 | ⚠️ 색상바에서 간접적으로만 | ⚠️ 색상바에서 간접적으로만 |
| 필압 변화 | ❌ 동영상에서는 보기 어려움 | ⚠️ 선 굵기로 반영(약하게) | ⚠️ 선 굵기로 반영(약하게) |
| LLM 호환성 | ❌ Gemini만 지원 | ✅ 모든 비전 LLM | ✅ 모든 비전 LLM |
| 처리 속도 | ❌ 느림 (동영상 인코딩/디코딩) | ✅ 빠름 (이미지 1장) | ✅ 빠름 (이미지 2장) |

### 난이도별 특성 분석

#### 초등 문제
- 풀이가 비교적 단순 (수식 1~2줄, 답 1줄)
- 동영상: 1분 내외, 토큰 ~17,000
- 이미지: 색상 변화가 적어 순서 파악이 쉬움
- **결론**: 이미지로도 충분히 채점 가능. 동영상 불필요.

#### 중등 문제
- 풀이가 복잡해지기 시작 (방정식 풀이, 도형 작도 등)
- 동영상: 1~2분, 토큰 ~17,000~32,500
- 이미지: 획이 많아질수록 색상 구분이 미세해질 수 있으나, 딜레이 마커로 주요 사고 지점 파악 가능
- **결론**: 이미지(B안)가 최적. 원본으로 내용 읽고, 메타데이터로 순서 파악.

#### 고등 문제
- 복잡한 증명, 다단계 풀이, 그래프 작도 등
- 동영상: 2~5분, 토큰 ~32,500~78,900
- 이미지: 획 수가 많아(200+) 색상 그라디언트가 촘촘해지지만, 큰 구간(빨강 영역 vs 파랑 영역)은 구분 가능
- 딜레이 마커가 사고 지점을 명확히 표시하므로 핵심 정보 손실 없음
- **결론**: 이미지(B안) 권장. 토큰 절감 효과가 16~23배로 가장 극적. 동영상은 비용 대비 추가 정보가 크지 않음.

### A안 vs B안 선택 가이드

| 상황 | 권장 |
|------|------|
| 비용이 최우선 (대량 채점) | **A안** — 토큰 최소, 1장으로 충분 |
| 채점 정확도가 최우선 | **B안** — 원본 가독성 보장, 내용 + 순서 분리 |
| LLM이 색상 필기를 잘 읽을 수 있는 경우 | **A안** — 별도 원본 불필요 |
| 복잡한 고등 문제 (획 200+) | **B안** — 색상이 촘촘해져도 원본에서 내용 확인 가능 |
| API 호출 1회당 이미지 수 제한이 있는 경우 | **A안** — 이미지 1장으로 해결 |
